# ДЗ Lesson 9 — Find the Breaking Point

## 1. Завдання і гіпотези

Була побудована RAG-система, базована на MS MARCO датасеті різного розміру (1k, 10k, 100k, 300k)

| Компонент    | Інструмент                                                  |
|--------------|-------------------------------------------------------------|
| Датасет      | MS MARCO Passage Ranking через `BeIR/msmarco` (HuggingFace) |
| Embeddings   | `BAAI/bge-small-en-v1.5`                                    |
| Vector index | numpy brute-force (baseline) → FAISS HNSW (для fix)         |
| Hybrid       | BM25 + dense + RRF (для fix)                                |
| Eval         | Recall@K на основі qrels, latency p95                       |

Гіпотези:
1. При scaling корпусу від 1K до 300K passages recall@1 деградує швидше за recall@10, бо релевантний документ не зникає з топ-10, а лише сповзає з першої позиції під тиском семантично близьких distractors. 
2. Latency зростає лінійно з розміром (brute-force O(N)) і стає bottleneck'ом раніше, ніж recall@10. 
3. Hybrid retrieval (BM25 + dense + RRF) компенсує деградацію recall@1, бо BM25 сильніший на rare terms, які dense embeddings розмивають у великому корпусі.

## 2. Результати

1. (доведена) Recall@1 деградує швидше ніж recall@10 (до фікса і після нього)

| Size | Naive Numpy<br/>recall@1 / @10 | Hybrid RRF<br/>recall@1 / @10 |
|-----:|-------------------------------:|------------------------------:|
| 1k   |                     0.98 / 1.0 |                   0.84 / 0.96 |
| 10k  |                    0.96 / 0.99 |                   0.76 / 0.96 |
| 100k |                     0.9 / 0.95 |                   0.61 / 0.87 |
| 300k |                    0.78 / 0.94 |                   0.51 / 0.79 |


2. (доведена) Latency зростає лінійно з розміром (brute-force O(N)) і стає bottleneck-ом раніше (між 10k та 100k) ніж recall@10 (між 100k та 300k)

| Size | Naive NumPy<br/>p95 (ms) |FAISS HNSW<br/>p95 (ms)  |
|-----:|-------------------------:|------------------------:|
| 1k   |                     0.06 |                    0.05 |
| 10k  |                     0.82 |                    0.14 |
| 100k |                    10.80 |                    0.32 |
| 300k |                    33.31 |                    0.33 |


3. (спростована) Hybrid retrieval (BM25 + dense + RRF) не покращує recall, а навпаки погіршує його

## 3. Розглянуті Fix-и

1. Для Latency: Заміна `Naive Numpy` на `FAISS HNSW`
   Latency зменшилася з 33ms до 0.33ms на 300k датасеті (~100 раз швидше) За графіком пошук змінився з лінійного на логарифмічний.

2. Для Recall: Додавання Hybrid retrieval (BM25 + dense + RRF) 
   Hybrid RRF показав 0.51 для recall@1 для 300k (в порівнянні з без нього - 0.78).
   Це може пояснюватися наступними факторами:
   - BM25 може додавати шумні збіги за ключовими словами, що може "збити" ранжування результатів.
   - набір даних може більше підходити для семантичного пошуку, ніж для лексичного
   - можливо, треба використовувати кращі методі токенізації (stemming, stopwords).

## 4. Рекомендації для датасету 1M+

1. **Vector index**: бажано використовувати HNSW, оскільки пошук по Naive буде займати >100ms (екстраполяція з 33ms на 300k → ~110ms на 1M, лінійно).


2. **Recall**: тренд показує recall@1 = 0.78 на 300k → екстраполяція дає ~0.65–0.70 на 1M. Для відновлення:
   - **Перейти на сильніший embedding** (bge-large-en-v1.5 / bge-m3) — це найбільший важіль, ~+5–10% recall@1.
   - **Cross-encoder reranking**: брати top-50 з HNSW, переранжувати через `bge-reranker-base`.
   - **Hybrid retrieval треба переробити**, перш ніж вмикати на проді: справжній BM25 (Elasticsearch/Tantivy/PISA замість Python `rank_bm25`)

3. **Sharding & distributed**: на ~10M+ один процес стає bottleneck. Розбити корпус на N шардів, паралельні `search()`, мердж результатів.
