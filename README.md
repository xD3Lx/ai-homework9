# ДЗ Lesson 9 — Find the Breaking Point

R&D-завдання: побудувати RAG-систему і **навмисно її зламати** через scaling, щоб знайти, де саме і чому вона починає деградувати.

## Гіпотеза (для перевірки)

> При scaling корпусу від 1K до 300K passages **recall@1 деградує швидше за recall@10**, бо релевантний документ не зникає з топ-10, а лише сповзає з першої позиції під тиском семантично близьких distractors. **Latency зростає лінійно** з розміром (brute-force `O(N)`) і стає bottleneck'ом раніше, ніж recall@10. **Hybrid retrieval (BM25 + dense + RRF)** компенсує деградацію recall@1, бо BM25 сильніший на rare terms, які dense embeddings розмивають у великому корпусі.

Ваша задача — **підтвердити або спростувати** цю гіпотезу на власних даних. Якщо ваші результати її спростовують — це **цінніший** результат, ніж підтвердження. Поясніть у звіті, чому ваші числа розійшлися з гіпотезою.

> 📌 Це наша робоча гіпотеза. Вона базується на загальних patterns з BEIR/MTEB literature, але не є істиною в останній інстанції. На MS MARCO специфічно: hybrid дає скромні +2-5% recall, тоді як reranker — +10-20%. Якщо ваш fix виявиться ефективнішим за hybrid (наприклад, reranker або кращий embedding) — обов'язково напишіть про це у звіті, це й буде ваш реальний R&D-внесок.

## Що треба зробити

1. Побудувати RAG pipeline (embeddings → vector index → retrieval → eval)
2. Прогнати його на **4 розмірах корпусу**: 1K → 10K → 100K → 300K passages
3. На кожному розмірі виміряти: Recall@1, Recall@10, MRR@10, latency p50/p95, RAM, embedding throughput
4. **Знайти точку перелому** — де метрики просіли на 20%+
5. Пояснити **чому** саме там і запропонувати fix
6. **Реалізувати fix** і виміряти, чи реально допоміг
7. Написати короткий звіт зі скріншотами і графіками

## Стек

| Компонент | Інструмент |
|---|---|
| Датасет | MS MARCO Passage Ranking через `BeIR/msmarco` (HuggingFace) |
| Embeddings | `BAAI/bge-small-en-v1.5` (швидко) або `BAAI/bge-m3` (точніше) |
| Vector index | numpy brute-force (baseline) → FAISS HNSW або Qdrant (для fix) |
| Reranker (для fix) | `BAAI/bge-reranker-v2-m3` |
| Sparse (для fix) | BM25 через `rank_bm25` |
| Eval | Recall@K, MRR@K на основі qrels — **без LLM**, $0 |

**Cost: $0** для основної частини. Якщо локально не тягне — див. секцію «Підказки» нижче.

## Метрики

Усі метрики обчислюються **без LLM** (порівняння IDs документів) — імплементація в `template/metrics.py`:

- **Recall@1** — чи правильний doc стоїть на 1-му місці. Найчутливіша до scaling.
- **Recall@10** — чи в топ-10 взагалі є правильний doc. Стеля якості.
- **MRR@10** — середня позиція релевантного doc. Інтегральний показник.
- **Latency p50/p95/p99** — на скільки повільнішає search
- **Embedding throughput** — passages/sec на вашому залізі
- **RAM** — пам'ять для embeddings + index
- **Disk** — розмір persisted index

## Scope

- Розміри: 1K → 10K → 100K → 300K
- **Реалізувати fix**: hybrid (BM25 + dense + RRF) АБО HNSW замість Flat
- Показати, що fix покращує метрики, які просіли
- Короткий звіт + скріншоти baseline vs after-fix

> 🚀 **За бажанням**: якщо є ресурси і ентузіазм — розширте до **500K → 750K → 1M passages**. На таких розмірах brute-force вже відверто ламається (latency >100ms, RAM >2GB), і саме тут видно, чому HNSW / disk-based індекси існують. Окремо в звіті позначте extrapolation vs реальний run і опишіть, які bottleneck'и побачили на 1M, яких не було на 300K.

## Підказки

### Якщо локально не вистачає ресурсів
Якщо ноутбук не тягне 100K-300K embeddings (мало RAM, повільний CPU, немає GPU) — **не блокуйтесь на цьому**. Використайте API:
- **OpenAI** `text-embedding-3-small` ($0.02 / 1M tokens) — 300K passages по ~100 токенів ≈ $0.60
- **Cohere** `embed-v4.0` ($0.10 / 1M tokens)
- **Voyage** `voyage-3-lite` ($0.02 / 1M tokens)
- **Jina** `jina-embeddings-v3` (free tier 1M tokens/місяць)

Повний run на 300K через API коштуватиме <$1, embedding generation займе хвилини замість годин. У звіті чесно вкажіть, який стек ви використали і чому. Це **не штрафує оцінку** — на enterprise scale API часто і є правильним вибором.
